# LLM training on ARC-AGI and other tasks
This repository contains experiments with autoregressive models for ARC-AGI and several other datasets.
It includes a reimplementation of GPT-2-style models with multiple architectural variations.
The same codebase is used for text pretraining, ARC-AGI training, and related reasoning tasks.

# Quickstart ARC-AGI Training
In this project, I reached 42% accuracy in TTA pass@2 (metric `acc_val_tta_pass2`) on the ARC-AGI 1 evaluation set with an 18.7M-parameter model.
For reference, the "Hierarchical Reasoning Model (HRM)" reports 40.3% on the same metric.
This suggests that a much simpler model with a more standard training setup can surpass HRM.
More recent papers have reported higher numbers, such as 54.5% in "ARC Is a Vision Problem!". Those results exploit the 2D structure of the data by treating ARC as a vision problem.
I plan to explore that direction in future work.

Below I explain the training setup, with a focus on the model configuration in `configs/arc_agi_l3x10_alllosses.json` and the corresponding training arguments in `configs/arc_agi_train_b2long_standalone.json` that were used to get that result.

To install:

```bash
pip install -r requirements.txt
```

To replicate:

```bash
export TOKENIZERS_PARALLELISM=true
python train.py --config configs/arc_agi_l3x10_alllosses.json --name_exp l3x10alllosses_long_standalone --args_from_json configs/arc_agi_train_b2long_standalone.json
```

The headline metric `acc_val_tta_pass2` assumes the default `--num_runs_accuracy 100`. If you change that flag, the logged key becomes `acc_val_tta_pass2_<N>aug`.

## ARC-AGI puzzle
ARC-AGI tasks are small abstract reasoning puzzles built from colored 2D grids. Each cell contains an integer color value, usually in `0..9`. A task does not come with a text description of the rule. Instead, the rule has to be inferred from examples.

Conceptually, one task looks like this:

1. Several example pairs are given.
2. Each example pair contains an input grid and the correct output grid.
3. Those examples are all generated by the same hidden transformation rule.
4. One or more additional input grids are provided.
5. For those additional inputs, the output grid must be predicted.

So the core ARC problem is: infer the transformation from a few input/output demonstrations, then apply it to a new input.

This is closer to program induction or rule induction than to ordinary language modeling. The sequence model in this repository is trained to imitate that behavior after the grids are serialized into text.

### JSON structure of a task

In the ARC files used by this repository, each task is represented as a dictionary with two main fields:

- `train`: a list of solved examples
- `test`: a list of test examples to solve

Each example itself contains:

- `input`: the source grid
- `output`: the output grid

The output of each example in `test` is what the model is trained to predict based on the preceding examples and the test input.

Schematic task structure:

```json
{
  "train": [
    {"input": [[...]], "output": [[...]]},
    {"input": [[...]], "output": [[...]]}
  ],
  "test": [
    {"input": [[...]], "output": [[...]]}
  ]
}
```

The training set is a list of tasks. Each task is a dictionary with `train` and `test` lists, and each example in those lists has an `input` grid and an `output` grid.
The validation set is structured the same way.

## How one ARC task becomes a sequence

The model does not consume 2D tensors directly in this setup. `arc_agi_dataset.py` converts each task into a single character sequence.

The logic is:

1. List the solved training examples first.
2. For each solved example, write the input grid and then its output grid.
3. After that, write each test input grid.
4. After each test input, append the corresponding output span that the model is trained to predict token by token.

So if a task contains:

- `train = [(input1, output1), (input2, output2)]`
- `test = [(input3 -> output3)]`

The serialized order is conceptually:

```text
[BOS]
input1, output1,
input2, output2,
input3, output3
```

The `test` list can also contain multiple input/output pairs.

The critical distinction is that the final `output3` belongs to the test example. It is the continuation the model is supposed to infer from the pattern shown in the preceding training pairs.

The model is autoregressive, so it generates one token at a time. The training loss is computed on the test output tokens (and possibly on all output tokens), but the model also sees the test input tokens as part of the context before it starts generating.
Training still runs on the whole sequence at once because Transformer-style attention lets the model attend to all previous tokens when generating each output token.

## ARC prompt format

The previous section described the puzzle semantics. This section summarizes the concrete token format used by the code.

Each task is serialized into a character-level prompt by `create_prompt()`.

`create_prompt()` builds the sequence with explicit markers:

- `[BOS]` is added at the front
- each input grid is prefixed by `I`
- each output grid is prefixed by `O`
- rows inside a grid are separated by `|`
- each grid ends with a newline `\n`

So one training pair and one test pair are serialized like this:

```text
[BOS]I<input grid>
O<output grid>
I<test input grid>
O<test output grid>
```

The format is therefore roughly:

```text
[BOS]I<input grid>O<output grid>I<input grid>O<output grid>...
```

A tiny toy example with 2x2 grids would look like:

```text
[BOS]I12|34
O56|78
I90|12
O34|56
```

In actual Python string form, that same prompt is closer to:

```python
"[BOS]I12|34\nO56|78\nI90|12\nO34|56\n"
```

This should be read as:

- first solved example: `I12|34 -> O56|78`
- second unsolved case to complete: `I90|12 -> O34|56`

During training and validation, the repository already knows the true final output, so it can compute loss or accuracy there. From the model's perspective, however, the job is still to continue the sequence correctly after the `O` marker of the test example.

The prompt uses the same `I...O...` surface format for both demonstration pairs and test cases. The difference is semantic rather than syntactic: the earlier pairs show the rule, while the final test input is the new case to solve, and its output tokens are present only so the repository can train on them or score predictions against them. There can also be multiple test examples.

Optional thinking tokens `T` can be inserted after inputs, but in this specific config `think_budget` stays at its default of `0`, so no `T` tokens are actually inserted.

During training, sequences are padded with `[PAD]` to a fixed length according to the configured sequence length.

### Tokenization

The ARC dataset uses `arc_agi_tokenizer.json` through `PreTrainedTokenizerFast`.

This setup is effectively a tiny character-level vocabulary for:

- digits `0` to `9`
- control symbols used by the prompt format, such as `I`, `O`, `T`, `|`, and newline `\n`
- special tokens `[UNK]`, `[BOS]`, `[EOS]`, `[PAD]`

That is why the model config uses `vocab_size=19`.

### Which parts are supervised

The sequence contains both context and targets.

For ARC, the important supervision is on output-grid tokens, not on the input-grid tokens. With this config:

- the training examples provide demonstrations of the rule
- the test example's input provides the new case to solve
- the test example's output is the main thing the model is trained to predict

Using `gt_labels_only=true` and `gt_all_out=true` together keeps output-content tokens as labels while masking non-target positions. In this setup, the loss is computed not only on the test outputs, but also on training-example outputs except for the first one. This gives more supervision than training only on the test outputs. If `gt_labels_only` is disabled, the model is instead trained on all non-padding tokens, including input tokens and optional thinking tokens.

The important code-level detail is that `create_prompt()` does not treat every output span identically:

- the first training example's output is excluded from `mask_all_out`
- later training example outputs are supervised
- all test outputs are supervised
- the `O` marker itself is not supervised; supervision starts on the first token after `O`

So train and test examples are still different in practice. The loss can include later train outputs because `gt_all_out=true`, but the ARC accuracy mask is defined on test outputs only, and if `repeat_test > 1` only the last repeated test output contributes to that accuracy mask.

So the prompt is not just a plain string. It is a structured sequence where some spans are context and some spans are designated target regions.

## How these configs are used

The training entrypoint is `train.py`.

To reproduce the main ARC-AGI result, run:

```bash
export TOKENIZERS_PARALLELISM=true
python train.py --config configs/arc_agi_l3x10_alllosses.json --name_exp l3x10alllosses_long_standalone --args_from_json configs/arc_agi_train_b2long_standalone.json
```

The flow is:

1. `train.py` loads the training arguments from `--args_from_json configs/arc_agi_train_b2long_standalone.json`, overriding some of the default arguments in `train.py`.
2. `config_name="custom:gptv0_standalone"` resolves through `custom_func.custom_config()` to `model_standalone.CustomGPTConfig`.
3. `--config configs/arc_agi_l3x10_alllosses.json` updates the model configuration before the model is instantiated.
4. `custom_func.custom_model()` instantiates `model_standalone.CustomGPTmodel`.
5. Because `dataset_name="custom:arc_agi"`, `train.py` uses the ARC-specific dataset branch instead of the Hugging Face `datasets` token-chunking path.

## Dataset used by this config

The ARC loader is `get_arc_agi_dataset()` in `arc_agi_dataset.py`.

It reads:

- `arc_agi/arc-agi_training-challenges.json` and `arc_agi/arc-agi_training-solutions.json`
- `arc_agi/arc-agi_evaluation-challenges.json` and `arc_agi/arc-agi_evaluation-solutions.json`
- `arc_agi/arc-agi_concept-challenges.json` and `arc_agi/arc-agi_concept-solutions.json`

Raw task counts in this repository are:

- training set: 400 tasks
- evaluation set: 400 tasks
- concept set: 160 tasks

With the provided dataset config:

```json
{
  "aug": true,
  "gt_labels_only": true,
  "include_val_train": true,
  "include_concept": true
}
```

the effective training corpus is assembled as follows:

- Start from the 400 official ARC training tasks.
- Add 400 extra tasks derived from the evaluation challenges when `include_val_train=true`.
- Add all 160 concept tasks as training when `include_concept=true`.

So the training split becomes effectively 960 tasks before any synthetic augmentation.

### What `include_val_train=true` actually does

This is not a simple copy of the evaluation split into training.

For each evaluation task, the loader builds an auxiliary training task where:

- `train = original evaluation train examples except the last one`
- `test = the last original evaluation train example`

So the model is exposed to the evaluation tasks' demonstration pairs during training, but not to the official evaluation test answers through this path.

This is not test-time training in the usual sense. In standard test-time training or test-time adaptation, the model is updated at evaluation time, after seeing a test task or test input. That is not what happens here.

Instead, `include_val_train=true` performs an offline dataset transformation before the normal training loop starts. It turns part of each evaluation task into an additional supervised training example and then trains the model on it together with the ordinary training set.

So the closest description is:

- not test-time training
- not use of the official evaluation test outputs
- but a transductive or evaluation-aware training setup, because part of the evaluation tasks is reused during training

This is therefore not cheating in the narrow sense of leaking the measured targets, because the actual evaluation outputs that the model is later scored on are not provided in the training data.

That distinction matters for interpretation, and the evaluation section below explains exactly what this repository uses as the ARC validation split.

### Labels and masks

Because `gt_labels_only=true`, the dataset masks out non-target tokens from the loss by setting their labels to `-100`.

With the current ARC prompt builder and default `gt_all_out=true`:

- input tokens are not supervised
- test output tokens are supervised
- train output tokens are also supervised, except for the first training example's output
- the `O` marker tokens are not supervised

This matches the logic in `create_prompt()` and `ArcAGI.__getitem__()`.

Because `aligned_labels=true`, the dataset shifts labels itself:

- `labels = labels[1:] + [-100]`
- `mask = mask[1:] + [0]`

This shifts the targets to predict the next token in the sequence, which is the standard autoregressive setup for language modeling.
That means the loss function receives already aligned labels and does not apply an extra next-token shift, unlike the Hugging Face transformers loss implementation.

### Sequence length and batching

The training config sets:

- `block_size=9400`
- `stride=9400`

The dataset pads each example to that length.

At batching time, `crop_not_valid=true` switches the collator to `crop_data_collator()` from `preprocess.py`, which trims the batch to the last position where any label is valid. This reduces wasted compute on right padding while preserving alignment across `input_ids`, `labels`, `mask`, and optional fields such as `position_idx` when present.

For this ARC setup, some auxiliary fields are only materialized when they are actually needed:

- `position_idx` (not used for the described config) is only built when the model enables `use_pos_emb_2d` or `use_rot_emb_2d`
- `mask_input` is only built for generate-based accuracy or TTA evaluation paths

## Data augmentation used for ARC

Data augmentation is a critical part of the training setup for ARC-AGI and related tasks, because the dataset is small and the model can easily overfit.

### Geometric and color augmentation

Each task can receive:

- a random color permutation
- a random geometric transform from the 8 dihedral symmetries
- optional 2x upscaling augmentation, transform id `8` (not used in this config because `max_aug_idx=7`)

By default, color remapping excludes color `0` (black), because `exclude_black=True` unless overridden.

### Shuffle augmentation

The augmentation code also supports a shuffle transform (transform id `9`) that permutes flattened grid cells.

That path is controlled by `shuffle_prob`.

In this config, `shuffle_prob` is not set, so it stays at the dataset default of `0.0`. That means the shuffle transform is available in code but not active for the provided training config.

### Input/output transform coupling

The code can apply different transforms to inputs and outputs when `aug_inp_out=True`.

That is not enabled here, so inputs and outputs receive the same transform and color mapping.

### Example order shuffling and IO swapping

`apply_augmentation()` also changes task structure:

- if `shuffle_examples=True`, with probability 0.5 the code concatenates train and test, shuffles everything together, and then splits back to the original counts; otherwise, it shuffles train and test within their own splits
- that means examples can move between the train and test portions of the serialized task while preserving the number of train and test examples
- with probability 0.4, inputs and outputs are swapped for every example in the task (so instead of predicting the output from the input, the model tries to predict the input from the output)
- if full shuffling is not used, `shuffle_train_examples=True` allows train-only shuffling

The defaults in `ArcAGI` are:

- `shuffle_examples=True`
- `shuffle_train_examples=True`

So these behaviors are active for the provided config.

### Train-example reduction

Another augmentation-like regularizer is `reduce_train=True`, which is enabled by default.
With probability `0.3`, the code keeps only a random prefix of the task's training examples:

```python
task["train"] = task["train"][: random.randint(1, num_train)]
```

This makes the same ARC task appear with varying context sizes across epochs.
If `shuffle_examples` is enabled, `reduce_train` may be less important as a data augmentation.

### Features supported by the dataset but not used here

These exist in the dataset code but are not active in the provided config:

- `num_synthetic_data > 0`: append synthetic identity-style tasks
- `noise_prob > 0`: replace some supervised tokens with random in-sequence noise
- `use_unk_token_for_gt=True`: replace supervised input positions with `[UNK]`
- `repeat_test > 1`: repeat test examples inside the prompt
- `think_budget > 0`: insert `T` tokens after inputs

## Model used by this config

The model is the standalone transformer defined in `model_standalone.py`.

Resolved configuration from `arc_agi_l3x10_alllosses.json`:

- `vocab_size=19`
- `context_len=9400`
- `embed_size=720`
- `num_layers=3`
- `heads=12`
- `dropout=0.1`
- `selfatt_class="SelfAttentionV2"`
- `rotary_emb=32`
- `repeat_model=10`
- `random_repeat=false`
- `loss="StandardLoss"`
- `reduce_loss="sum"`
- `aligned_labels=true`
- `curriculum=false`
- `all_repeat_losses=true`
- `tie_word_embeddings=false`

### High-level architecture

At a high level, this model consists of:

- token embeddings
- rotary positional encoding `rotary_emb=32`
- 3 transformer-style blocks of class `SelfAttentionV2`
- final normalization
- linear LM head

`SelfAttentionV2` itself is a causal pre-norm block with:

- RMSNorm before attention
- multi-head causal self-attention
- an `MLPv2` feed-forward branch
- learnable residual scaling parameters `gamma_attn` and `gamma_mlp`
- dropout on attention and residual paths
- flash attention through `torch.nn.functional.scaled_dot_product_attention()` when enabled

### What `repeat_model=10` means here
When `repeat_model > 1`, the code does not duplicate the whole 3-layer stack in the obvious way. Instead it uses:

- the first block once
- the middle block repeated `repeat_model` times
- the last block once

Because `num_layers=3`, there is only one middle block.

So this config behaves like:

```text
block0 -> block1 -> block1 -> block1 -> ... (10 times total) -> block2
```

That is a shallow base architecture with a deep repeated middle computation.
This allows the model to increase computational depth without increasing the number of unique parameters, because the same block is reused. That reduces the risk of overfitting on a small dataset while still giving the model more capacity to perform iterative computations.

### What `all_repeat_losses=true` does here

The proposed config asks the model to compute auxiliary losses on later repeated states.
Here, the number of repeats is fixed and the curriculum is disabled, so the auxiliary losses are computed on a fixed set of repeats.

With `repeat_model=10`, `curriculum=false`, and `random_repeat=false`, `min_max_repeat_curriculum()` returns:

- `min_repeat = 5`
- `max_repeat = 10`

Since the curriculum is disabled, the number of repeats is always `max_repeat`. `min_repeat` controls when auxiliary losses start being collected, so in this case repeats 6 through 10 contribute an auxiliary branch.
Because `num_layers=3`, there is a single middle block being repeated. The effective training graph is therefore:

```text
block0
-> middle repeat 1
-> middle repeat 2
-> middle repeat 3
-> middle repeat 4
-> middle repeat 5
-> middle repeat 6  --side branch--> block2 -> auxiliary loss
-> middle repeat 7  --side branch--> block2 -> auxiliary loss
-> middle repeat 8  --side branch--> block2 -> auxiliary loss
-> middle repeat 9  --side branch--> block2 -> auxiliary loss
-> middle repeat 10 --side branch--> block2 -> auxiliary loss
-> block2
-> main loss
```

More precisely, after repeats 6 through 10 the code takes the current hidden state, runs the final block (`block2`) plus norm and LM head on a side branch, and stores those auxiliary logits. The main forward path still continues through the remaining repeated middle blocks and only applies the final `block2` once at the end. Those auxiliary losses are averaged across the selected repeats, then combined with the final loss as:

```text
final_loss = 0.5 * main_loss + 0.5 * mean(auxiliary_repeat_losses)
```

So the model is not supervised only at the very end of the repeated computation. The later repeated states are also pushed to become predictive before the final pass.

### Weight tying and initialization

Two more details matter:

- `tie_word_embeddings=false` means input embeddings and output projection are separate matrices. (Since the vocabulary is small, tying is not critical for parameter efficiency.)
- `megatron_init=true` applies smaller initialization to residual projection layers using a GPT-2 / Megatron-style scaling rule.

## Loss and optimization details

### Loss

The selected loss is `StandardLoss` from `loss.py`.

Because `aligned_labels=true`, `StandardLoss` does **not** apply its internal next-token shift. It uses labels as provided by the dataset.

`StandardLoss` computes:

- cross-entropy over valid tokens only
- `ratio_correct` on valid tokens
- `num_valid` so logging can weight metrics by how many supervised tokens were present

### `reduce_loss="sum"`

This config uses `reduce_loss="sum"`, but the training loop applies an extra normalization step.

In `model_standalone.CustomGPTmodel.forward()`:

1. the token loss is computed in summed form
2. if repeat losses are present, they are averaged with the main loss
3. during training, the result is finally scaled by:

```text
1 / context_len / batch_size
```

So the effective training loss is a manually normalized summed loss, not a plain mean reduction inside the loss function itself.
This seems to help with training stability in this setup, but I have not done systematic ablations on that detail.

### Optimizer

The optimizer is created by `create_optimizer()` in `scheduler.py`.

For this config it is:

- `AdamW`
- learning rate `6e-5`
- `beta1=0.9`
- `beta2=0.95` (hard-coded in the training script)
- weight decay `0.45` (high because the dataset is small and the model can easily overfit)

Parameter grouping is important:

- matrix-like parameters get weight decay
- parameters with dimension `<= 1` do not get weight decay (biases, layer norm gains, and residual scaling parameters)
- special handling exists for `LowRankLinear`, but this model config does not request low-rank layers

### Scheduler

The scheduler is created by `create_lr_scheduler()`.

This config uses:

- `lr_scheduler_type="cosine_with_min_lr"`
- `num_warmup_steps=2000`
- `min_learning_rate=1e-7`

That goes through Hugging Face scheduler creation, with `min_lr` passed as a scheduler-specific argument.

### Precision, compilation, and clipping

The training config also sets:

- `precision="bf16"`
- `grad_clip_value=1.0`
- `compile=true`

So the model is compiled with `torch.compile()` before `accelerator.prepare()`, and gradients are clipped to a norm of `1.0` when synchronized.

## Training-loop behavior for this setup

### Batch sizes

The provided training JSON sets:

- `per_device_train_batch_size=2`
- `gradient_accumulation_steps=1`
- `per_device_eval_batch_size=32`

The effective total train batch size is:

```text
per_device_train_batch_size * num_processes * gradient_accumulation_steps
```

Assuming a single GPU, the total batch size is `2`.

### Dataloaders

For ARC-AGI, `train.py` uses `num_workers=6`.

Because `crop_not_valid=true`, both train and validation dataloaders use `crop_data_collator()` instead of the default Hugging Face collator.


## Evaluation for ARC

ARC uses a custom evaluation branch in `evaluation.py`.

The main ARC metrics are:

- `acc_train` on the training split
- `acc_val` on the validation split
- `acc_val_tta_pass2` on a dedicated test-time-augmentation dataset
- in `debug` mode, `acc_train_tta_pass2` on a dedicated train-side TTA dataset

In `debug` mode, the code also computes `acc_train_generate` plus two validation-side TTA variants with 10 augmentations (`strict=True` and `strict=False`).

### What the validation split actually is

For ARC in this repository, validation is not a random held-out subset of the training split.

Validation uses the official ARC evaluation tasks with solutions attached, loaded from:

- `arc_agi/arc-agi_evaluation-challenges.json`
- `arc_agi/arc-agi_evaluation-solutions.json`

When `include_val_train=true`, validation here is not a completely clean held-out estimate of generalization to unseen tasks, because the model has already trained on demonstrations from those same evaluation tasks. Nonetheless, the official evaluation test outputs themselves are still not used as training targets.

### Which dataset each metric uses

The evaluation code does not use a single dataset for everything.

- `acc_train` uses the ordinary training dataloader. With this config, that training dataset still has `aug=True`, so `acc_train` is measured on augmented training views rather than on a fixed canonical train split.
- `acc_val` uses the ordinary validation dataset, where `aug=False`
- `acc_train_tta_pass2` uses a separate `train_test` dataset and is only computed in `debug` mode. Its augmentation flag is built as `aug=not args.debug`, so in debug runs it is mainly a train-side sanity check rather than a headline metric.
- `acc_val_tta_pass2` uses a separate `validation_test` dataset, where `aug=True`

Accuracy on training is only for debugging purposes. The main validation metric is `acc_val_tta_pass2`, which is a TTA-based pass@2 metric computed on augmented views of the official evaluation tasks.

More precisely, `train.py` uses three ARC datasets at evaluation time:

- `train_test`: augmented prompts for train-side TTA/pass@2 evaluation
- `validation`: unaugmented prompts for the standard validation metric
- `validation_test`: augmented prompts for TTA/pass@2 evaluation

The TTA datasets are built with additional constraints so that predictions can be compared across augmentations and inverted back to the original task space. For `validation_test`, and for `train_test` when `debug=false`, that configuration includes:

- `aug=True`
- `num_synthetic_data=0`
- `compute_mask_input=True`
- `shuffle_examples=False`
- `reduce_train=False`
- `max_aug_idx=7`
- `shuffle_prob=0.0`
- `rand_think=False`
- `repeat_test=1`
- `return_aug_params=True`

Inside `compute_acc_tta_pass2()`, the evaluator repeatedly calls `dataset[idx]` to sample multiple augmented views of the same underlying ARC task, generates predictions for each view, inverts the augmentations back to the original task space, and then performs top-2 voting. That is why pass@2 is augmentation-based while the standard validation metric is not.
By default, `num_runs_accuracy=100`, so the repository's main validation TTA metric is effectively a pass@2 evaluation over 100 augmentations. If you set a different value, the reported metric key becomes `acc_val_tta_pass2_<N>aug`. HRM uses 1,000 augmentations, but that is slower; I plan to test it in the future.

## Practical summary of this specific configuration

This setup is best understood as:

- a character-level ARC sequence model
- trained on 960 ARC-family tasks assembled from training, evaluation-derived, and concept splits
- using output-only supervision (`gt_labels_only=true` and `gt_all_out=true`), so the model is supervised on later output tokens from both train and test examples, not only on the test outputs. The main exception is the first training example's output.
- with strong geometric/color/task-structure augmentation on training examples
- with a 3-block standalone transformer where the middle block is repeated 10 times
- with auxiliary losses on later repeated states
- using bf16, AdamW, cosine decay
- high weight decay and dropout to regularize the model on the small dataset
- a loss that is manually normalized by context length and batch size rather than a plain mean reduction. This should improve training, but a more systematic comparison of `reduce_loss="sum"` vs `reduce_loss="mean"` is left for future work.
